### Header

- `header` component. Sticky nav: wordmark + logo left, "View demo" (ghost `button`) + "Get started" (primary `button`) right.
- No nav links вҖ” page is short enough to scroll without an anchor menu. Two CTAs serve two visitor intents: explore first vs. convert now.
- No login link вҖ” returning users know where `/auth` is; the header stays clean for first-time visitors.

### Hero

- `hero-02` вҖ” centered layout. Bold headline "Know your team." with a two-line subtitle. Two CTA buttons below: primary "Get started вҶ’" and ghost "View demo".
- App mockup of the directory list inside a `mockups` browser chrome wrapper вҖ” shows avatars, name, role, tenure at a glance. Proves the product before scrolling.
- No `text-rotator` вҖ” the headline is direct and doesn't need a cycling word. Rotating words undermine the confident, no-jargon voice.

### Feature showcase

- `feature-showcase-02` with `cycler-nav` (auto-cycling tabs). Four tabs: Directory, Profiles, Birthdays, Overview. Each tab shows a product screenshot inside `mockups` browser chrome.
- Auto-cycling communicates the full product surface without requiring the visitor to scroll past four static blocks.
- No static `feature-showcase-01` grid вҖ” the directory's value is in the interaction pattern (grouped rows, avatars), which a cropped static grid thumbnail can't convey.

### Why teams love it

- `features-03` вҖ” three-column icon-led benefit cards: [Users] Know your team, [Cake] Never miss a birthday, [Handshake] Grow together.
- Lucide icons only: `Users`, `Cake`, `Handshake`.
- No feature grid duplication вҖ” this section communicates human outcomes, not features. The showcase above handles feature specifics.

### Testimonials

- `testimonial-01` вҖ” horizontal carousel, center card highlighted, flanking cards fade and shrink. Three testimonials from team leads and ops managers.
- No wall of logos вҖ” the audience is small-team operators, not enterprise buyers. Peer quotes land better than brand names.
- Dots pagination below. Three entries is enough; a slider would feel excessive.

### CTA banner

- `cta-01`. Single closing moment before the footer. Repeats "Get started вҶ’" вҖ” same destination as the hero. No second CTA here вҖ” one decision, one button.

### Footer

- **new** `pages/landing/components/footer.tsx` вҖ” three-column minimal footer: logo + tagline left, Product/Company links center, Legal links right.
- No newsletter form вҖ” the product converts on sign-up, not email capture.
- No social icons row вҖ” this is a B2B team tool; social proof is handled by testimonials above.

### Page shell

- `application-layout` вҖ” fullscreen, no topbar chrome. Centered card on a muted background.
- "вҶҗ Home" ghost link top-right returns to `/`. Back link rather than a full header keeps the page focused on the single job: authenticate.
- No sidebar, no topbar nav вҖ” auth is a focus-mode screen. Zero distraction.

### Auth card

- shadcn `card` with `tabs` for Sign in / Sign up toggle. Tab switch updates the headline copy and shows/hides the "Forgot password?" link.
- OAuth buttons: two full-width `button` (outline variant) with SVG brand icons вҖ” Google `[G]`, Apple `[A]`. OAuth first because it's the lowest-friction path.
- Divider: "or" text separator between OAuth and email form.
- Email + password: shadcn `form` with two `input` fields and validation. Primary `button` "Sign in" / "Create account" at bottom.
- Sign up tab adds a "Full name" field above email. No confirm-password field вҖ” reduces friction; email-based recovery handles the mistake case.
- On success: both OAuth and email/password navigate to `/overview`.
- No "remember me" checkbox вҖ” session management is handled server-side. Adding it would imply a choice that doesn't meaningfully exist.

### Topbar

- Same `workspace-layout-02` topbar as Overview. "+ New" primary `button` navigates to `/directory/new`.

### Toolbar (iCloud-style grouped actions)

- Sticky toolbar below the topbar, above the page header. Three action groups separated by flex spacers:
  - **Left group**: `[вүЎ]`/`[вҠһ]` view toggle (list/grid, `toggle-group`). No sort here вҖ” sort lives in the page header "by First name в–ҫ" control.
  - **Center group**: `[+ New]` primary creation button вҖ” navigates to `/directory/new` (full-page create form). `[вҶ‘ Import]` ghost button вҖ” opens CSV Import `dialog`. Two creation paths: one-at-a-time and bulk.
  - **Right group**: `[рҹ—‘]` delete (ghost, disabled until вүҘ1 row selected) + `[В·В·В·]` more actions dropdown.
- Search lives in the filter bar below (always visible), not in the toolbar вҖ” a people directory needs the search input ready to type at all times, not collapsed behind an icon.
- All actions immediately visible вҖ” no hidden menus for primary actions. `gap-2` within groups, `flex-1` spacers between groups.

### Page header (shared `PageHeader` component)

- `[Users]` icon + "Directory" heading left. "by First name в–ҫ" sort control right (`text-primary text-sm`) вҖ” clicking opens a `dropdown-menu` with options: First name AвҖ“Z (default), Last name AвҖ“Z, Start date (newest), Start date (oldest), Date added (newest).
- Subtitle: "10 employees В· 4 departments" (`text-muted-foreground`). Updates reactively when filters are active.
- Same `PageHeader` component as Overview, Calendar, Employee Detail вҖ” consistent across all screens.

### Filter bar

- Below page header: search `input` ("Search by name, role, or locationвҖҰ") + Department `select` + Location `select`. All in one row.
- Search filters in-place as the user types вҖ” matches against `full_name`, `role`, and `location` fields.
- Department options: All, Engineering, Product, Design, Operations (hardcoded enum).
- Location options: derived from unique `location` values in the dataset (All, SF, NYC, Austin, Remote).
- All filters stack вҖ” search + department + location can be active simultaneously.
- No date-range filter вҖ” this is a people list, not a time-series dataset.

### Column headers

- Thin header row below the filter bar: NAME, ROLE, DEPT, JOINED вҖ” `text-xs uppercase tracking-wider text-muted-foreground`. Dotted `border-b` below.
- Not sortable via column click вҖ” sort lives in the page hero and toolbar. Headers provide visual structure for scanning.

### Employee table (list view)

- shadcn `table` with department-grouped rows and dotted `separator` dividers between rows. Column headers: checkbox (select all), NAME вҶ‘ (sortable), ROLE, DEPT, JOINED.
- Department group headers in mono uppercase ("ENGINEERING В· 4") span full row width, styled with `text-xs font-medium uppercase tracking-wider text-muted-foreground`.
- Each row: shadcn `checkbox` left for multi-select + `avatar` (initials, 40px) + full name (semibold) + location in muted-foreground below the name + role + department `badge` (mint tint for engineering, other tints for other depts) + start year.
- Row end: `[вӢҜ]` `dropdown-menu` with "Edit" (navigates to `/directory/:id` in edit mode) and "Delete" (opens Confirm Delete `dialog`).
- Clicking the row (not the checkbox, not the overflow menu) navigates to `/directory/:id`.
- Hover: `bg-muted/60` wash on the row.
- No email/phone column вҖ” too much information for a scannable list. Contact details live on the profile. The table's job is identification, not contact lookup.

### Topbar

- Same `workspace-layout-02` topbar. "+ New" navigates to `/directory/new`.

### Page header (shared `PageHeader` component)

- `[Calendar]` icon + "Birthdays" heading left. "Month в–ҫ" view control right (`text-primary text-sm`).
- Subtitle: "3 upcoming" (`text-muted-foreground`). Updates based on the currently viewed period.
- Same `PageHeader` component as all other screens.

### Calendar controls

- View toggle: shadcn `toggle-group` with three options вҖ” Month, Week, Year. Month view is the default and most useful for at-a-glance birthday awareness.
- Previous / Next navigation: `[ChevronLeft]` and `[ChevronRight]` `button` (ghost) flanking the current period label "January 2025".
- Controls sit in a single row above the calendar grid.
- No "Today" button вҖ” adding it creates a tertiary control for a low-frequency action. The user can click next/prev to get back. Add if user research shows it's needed.

### Calendar grid (month view)

- shadcn `calendar` with custom day cell renderer. Days of the week as column headers (SunвҖ“Sat).
- Days that have a birthday: show a circular `avatar` dot (32px, initials) below the date number. Avatar uses the same initials + color system as the directory.
- Multiple birthdays on the same date: stack up to 2 avatar dots, "+N more" text if there are more than 2.
- Clicking an avatar dot navigates to `/directory/:id` for that employee.
- Today's date is highlighted with a subtle ring (shadcn `calendar` default today indicator).
- No event titles next to dots вҖ” just the avatar. The identity is the event. Text would crowd the cell.
- Weekends are not dimmed вҖ” birthdays fall on any day and all days should feel equally actionable.

### Calendar grid (week view)

- Seven-column week grid showing the current week (SunвҖ“Sat). Each column shows the date number at the top and any birthday avatars below.
- Same click-to-profile behavior.
- No time slots вҖ” birthdays are all-day events. A time-slotted week view (like Google Calendar) would be wrong for this use case.

### Calendar grid (year view)

- 12-month mini grid. Each month is a compact 7Г—5 cell grid showing only date numbers. Dates with birthdays get a small mint-colored dot indicator (no avatar вҖ” too small to render legibly).
- Click a date navigates to the month view for that month (not directly to a profile вҖ” at this zoom level multiple birthdays could share a date).
- Year view gives the "how spread out are our birthdays?" overview at a glance.

### This month panel

- Below the calendar grid: a row listing all birthdays in the currently viewed month. Format: `avatar` (32px) + "Full Name (Month Day)". Separator line above.
- Empty month: "No birthdays this month." in muted-foreground.
- Clicking a name navigates to `/directory/:id`.
- No separate sidebar panel for the birthdays list вҖ” keeping it inline below the calendar avoids a split-pane layout for what is already a single-focus screen.

* * *

**Data shape** (`src/data/seed.ts`):

- Birthdays derived from `employees.birthday` ("MM-DD") field вҖ” no separate entity needed.
- 10 birthdays across 10 employees ensure every month has at least one dot (Jan: 2, Feb: 1, Mar: 1, May: 1, Jun: 1, Jul: 1, Aug: 1, Sep: 1, Nov: 1).

**Data operations**:

| Operation | On this screen |
| --- | --- |
| **Create** | "+ New" topbar button вҶ’ `/directory/new` |
| **Read** | employees.birthday + employees.full_name + employees.id for all employees; rendered into calendar day cells |
| **Update** | none |
| **Delete** | none |
| **Filter** | month/week/year view toggle; prev/next navigation changes the rendered period |
| **Sort** | birthdays sorted by day-of-month within each month panel |
| **Search** | none |
| **Paginate** | none |
| **Aggregate** | birthday count per month for the "This month" panel |
| **Drill down** | click avatar dot вҶ’ Employee Detail (`/directory/:id`) |
| **Auth** | protected |

### Team tab

- Two sections: current members list and invite form.
- Members list: each row shows `avatar` (initials, 40px) + name/email + role `badge` (Admin = mint, Member = gray outline) + `[вӢҜ]` `dropdown-menu`.

- Pending invites show in the same list with a dashed `avatar` placeholder and "(pending)" label in muted-foreground.
- Invite form below the list: email `input` + role `select` (Admin / Member) + "Send invite" `button`. Submitted invite appears in the pending list in-place.
- No bulk role change вҖ” at small-team scale, individual row actions are sufficient and safer.

- Two settings: theme toggle and delete account.
